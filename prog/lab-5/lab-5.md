# Лабораторная работа №5. RPC. gRPC. Protobuf

## Задание

На основе продемонстрированного примера и мини-проекта с реализацией двух сервисов (рекомендация книг), реализуйте приложение, аналогичное приложение из задания 4, с использованием gRPC, protobuf.  Предоставьте ссылку на репозиторий GitHub со всеми необходимыми компонентами для развертывания. При возможности, разверните словарь на публичном сервере в вебе. 

В репозитории отразите отчет с помощью файла с разметкой Markdown, где демонстрировался бы процесс развертывания и работы сервиса.

## Решение:

Приложение из **двух gRPC-сервисов** по образцу демо «библиотека + рекомендации»,
для глоссария терминов **Python**:

* **GlossaryService** — хранит термины в **SQLite** (авто-миграция и наполнение
  при старте), реализует пять операций CRUD.
* **RecommendationService** — второй сервис, который сам является **gRPC-клиентом**
  первого: получает термины и формирует рекомендации.

Контракты описаны в **protobuf**, развёртывание — через **Docker Compose**.

## Архитектура

```
            ┌──────────────────────┐        ┌──────────────────────────┐
   client ─▶│   GlossaryService    │◀─ gRPC ─│  RecommendationService   │◀─ client
            │   :50051             │         │  :50052                  │
            │   SQLite (CRUD)      │         │  клиент GlossaryService  │
            └──────────────────────┘         └──────────────────────────┘
```

RecommendationService не хранит данные — на каждый запрос он по gRPC вызывает
`ListTerms` у GlossaryService и формирует ответ. Это и есть взаимодействие двух
сервисов из примера.

## Запуск

```bash
docker compose up --build
```

Поднимутся два сервиса: `glossary` на порту `50051` и `recommendation` на `50052`.
gRPC-код из `.proto` генерируется автоматически при сборке образа, таблица БД
создаётся и наполняется при старте `glossary`.

Запуск демо-клиента в новой другой cmd:

```bash
docker compose run --rm client
```
